# Unit 3 – Proposal, System Design and Project Planning

## Overview

Unit 3 focused on the design stage of the team project.  
During this unit, we were required to create a proposal and design document for the secure distributed system that would be developed later in the module. The aim of this stage was to plan the architecture, identify possible security risks, and decide which methods would be used to implement and test the system.

This unit showed that secure systems must be designed carefully before implementation. The design document helped the team understand the system requirements and define the responsibilities of each member.

---

## Project Proposal and Design

In this unit, our team discussed different ideas for the project and selected a system that could demonstrate secure distributed communication. We analysed the system using modelling techniques and identified possible vulnerabilities.

The proposal included:

- System architecture
- Threat analysis
- Security requirements
- Planned implementation methods
- Testing strategy

We also discussed how the system should handle unreliable communication, message loss, and possible attacks. This was important because the final project required us to simulate a distributed system.

The design stage helped us understand the importance of planning before coding.

---

## Life Cycle Methodology

During this unit, we learned that software development should follow a life cycle.  
Our team used an iterative approach, where we:

1. Designed the system
2. Implemented part of the system
3. Tested the system
4. Improved the design

This approach was useful because some problems only appeared after testing.  
For example, during the design we assumed that message authentication would be enough, but later we realised that replay attacks were still possible.

This showed that the development process must allow changes and improvements.

## Reflection

Unit 3 helped me understand that good design makes implementation easier.  
When the design is clear, the coding stage becomes more organised.

However, I also learned that the first design is not always correct. During later units, we had to change parts of the system after testing showed new problems.

This experience helped me understand why secure systems require both modelling and testing.

I also learned that working in a team requires clear communication. When we shared ideas early, the project progressed more smoothly.
